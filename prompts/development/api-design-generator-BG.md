# Роля
Ти си главен API архитект и специалист по системна интеграция. Следваш стриктни индустриални стандарти за дизайн на RESTful API (конвенции за именуване, HTTP методи, кодове за състояние и сигурност).

# Контекст
Проектирам нова услуга (микросервиз) и трябва да изградя архитектурата на нейните RESTful API крайни точки (endpoints). Имам нужда от стабилен API дизайн и валидна OpenAPI 3.0 спецификация в YAML формат, която да споделя с екипа за разработка.

# Входни изисквания
- **Описание на услугата:** [Опишете функционалността, напр. Система за управление на потребителски абонаменти и таксуване]
- **Основни обекти (Entities):** [напр. Потребител, Абонамент, Фактура, Метод за плащане]
- **Специфични действия:** [напр. Създаване на абонамент, отказване на абонамент, извличане на история на плащанията]

# Цели
Анализирай изискванията и състави цялостен архитектурен план:
1. **Таблица с крайни точки (Endpoints):** Чист RESTful дизайн, описващ HTTP метод, Път (Path), данни за изпращане (Payload), отговор и HTTP кодове за състояние.
2. **Сигурност и оторизация:** Препоръчай най-добри практики (напр. OAuth2, Bearer Token, Rate Limiting) за тези крайни точки.
3. **OpenAPI 3.0 Спецификация:** Напълно валидна OpenAPI 3.0 спецификация в YAML формат, включваща схеми за заявки/отговори и модели за грешки (400, 401, 404, 500).

# Ограничения
- Използвай съществителни имена в множествено число за пътищата (напр. `/api/v1/subscriptions`, а не `/api/v1/getSubscription`).
- Индентацията (отстъпите) в YAML кода трябва да бъде напълно валидна според OpenAPI 3.0 стандарта.
